# LLM (Large Language Model, большая языковая модель)

## Основные концепции

| Концепция | Что это | Зачем инженеру
|--|--|--
| Token | Минимальная единица текста (часть слова, слово, пробел) | LLM считают токены, а не символы. API тарифицируется в токенах
| Context window | Сколько токенов модель может "видеть" за раз (4K-1M) | Определяет, сколько текста можно скормить модели за раз
| Embedding | Векторное представление текста (уже обсуждали) | Поиск, кластеризация, RAG
| Temperature | Параметр случайности (0 → детерминированно, 1 → креативно) | Контролирует "фантазию" модели
| Inference | Получение ответа от модели | Главная задача: сделать это быстро и дёшево

**Типичные задачи инженера с LLM**

```py
# 1. Вызов модели через API (99% задач)
import openai
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "Кратко объясни градиентный спуск"}],
    temperature=0.7,
    max_tokens=500
)

# 2. Создание embedding'ов
embedding = openai.Embedding.create(
    model="text-embedding-ada-002",
    input="Текст для векторизации"
)

# 3. Дообучение (fine-tuning) на своих данных
# openai.FineTuningJob.create(...)
```

### Token — единица измерения всего

LLM не понимает буквы или слова. Она понимает токены — числовые ID из словаря (обычно 50K-200K уникальных токенов).

```py
# Пример: как токенизируется фраза
"Привет, как дела?" → [12563, 15, 245, 8923, 30]
```

Важно: токен ≠ слово. Одно слово может быть 1-3 токенами:
* "hello" → 1 токен
* "unfortunately" → 3 токена ("un", "for", "tunately")
* "Привет" → 1-2 токена (зависит от токенизатора)

**Инженерные последствия**
| Аспект | Что нужно знать
|--|--
| API тарификация | OpenAI, Anthropic и другие считают токены, не символы или запросы
| Context window | Измеряется в токенах (например, GPT-4 Turbo — 128K токенов)
| Скорость инференса | Модель генерирует токен за токеном (авторегрессивно)
| Лимиты | Каждый запрос/ответ ограничен по токенам (max_tokens)

**Как считать токены**

```py
import tiktoken  # библиотека от OpenAI

# Для GPT-4
enc = tiktoken.encoding_for_model("gpt-4")
text = "Привет, как дела? Это очень длинное сообщение..."
tokens = enc.encode(text)
print(f"Токенов: {len(tokens)}")  # не совпадает с длиной строки!

# Для ручного контроля в API
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[...],
    max_tokens=500,  # не даём сгенерировать больше 500 токенов
    temperature=0.7
)
```

* Для русского текста токенов примерно в 1.5-2 раза больше символов
* 1K токенов ≈ 750 слов английского, ≈ 400 слов русского
* Разные модели используют разные токенизаторы (GPT → cl100k_base, Llama → sentencepiece)

### Context window — "оперативная память" модели

Максимальное количество токенов, которое модель может "видеть" одновременно (вход + выход).

`[Входные токены] + [Сгенерированные токены] ≤ Context Window`

**Типичные размеры (2025 год)**

| Модель | Context Window
|--|--
| GPT-3.5 | 16K
| GPT-4 Turbo | 128K
| GPT-4.5 / o1 | 200K
| Claude 3 Opus | 200K
| Llama 3 (8B/70B) | 8K / 128K
| Mistral 7B v0.3 | 32K
| Qwen 2.5 | 128K

**Инженерные проблемы при большом контексте**

Проблема 1: Квадратичная сложность (влияет на скорость)
* Внимание (attention) работает за O(n²) по памяти, где n — длина контекста
* При 100K токенов модель может генерировать 1 токен за 2-5 секунд

Проблема 2: "Lost in the middle"
* Модель лучше помнит начало и конец контекста, середину "забывает"
* Решение: класть важные документы в начало или конец промпта

Проблема 3: Стоимость
* API считают все токены в контексте (и входные, и выходные)
* 128K токенов на входе → дорого

### Embedding — "смысл" в виде вектора

Embedding — это плотный вектор фиксированной размерности (обычно 768, 1024, 1536, 4096), который кодирует семантический смысл текста.

```
# Пример embedding'а (условно)
"кошка"   → [0.23, -0.45, 0.78, 0.01, ..., -0.32]  # размерность 1536
"кот"     → [0.24, -0.44, 0.77, 0.01, ..., -0.31]  # почти такой же
"собака"  → [0.45, -0.12, 0.56, -0.23, ..., 0.11]  # отличается
"ядерный" → [-0.89, 0.34, -0.12, 0.67, ..., 0.45]  # совсем другой
```

**Отличие от токенов**

| Токен | Embedding
|--|--
| Целое число (ID) | Вектор float'ов
| Только для конкретного слова/субслова | Есть у любого текста (через модель)
| Не несёт семантики | Кодирует смысл
| Используется внутри LLM | Используется для поиска, кластеризации

**Главное применение для инженера: семантический поиск**

```py
# 1. Создаём embedding'и для всех документов
documents = ["Документ 1...", "Документ 2...", ...]
doc_embeddings = embed_model.encode(documents)

# 2. Для запроса делаем embedding
query = "Как работает градиентный спуск?"
query_embedding = embed_model.encode(query)

# 3. Ищем ближайшие (косинусное расстояние)
from sklearn.metrics.pairwise import cosine_similarity
similarities = cosine_similarity([query_embedding], doc_embeddings)[0]
top_k_idx = similarities.argsort()[-5:][::-1]  # топ-5 похожих

# 4. Отдаём в LLM как контекст
relevant_docs = [documents[i] for i in top_k_idx]
```

**Важно**: embedding'ы разных моделей несовместимы (нельзя искать OpenAI вектором в базе, заполненной BGE).

### Temperature — управление случайностью

Параметр, который контролирует, насколько уверенно модель выбирает самый вероятный следующий токен.

Как работает:
Модель для следующего токена выдаёт вероятности (например, "кошка" — 60%, "собака" — 25%, "птица" — 15%).

| Temperature | Что происходит
|--|--
| 0 (минимальная) | Всегда выбирает самый вероятный токен ("кошка") → детерминированно, скучно
| 0.5 | Иногда выбирает второй-третий варианты, но редко
| 1.0 (стандарт) | Выбирает пропорционально вероятностям (60% "кошка", 25% "собака", 15% "птица")
| 2.0 | Почти равные вероятности, случайный бред

**Инженерное применение**

```py
# Кейс 1: Классификация (нужен один точный ответ)
response = openai.ChatCompletion.create(
    model="gpt-4",
    temperature=0.0,  # детерминированно
    messages=[{"role": "user", "content": "Классифицируй текст: {text}"}]
)

# Кейс 2: Генерация креатива (реклама, сценарии)
response = openai.ChatCompletion.create(
    model="gpt-4",
    temperature=1.0,  # разнообразие
    messages=[{"role": "user", "content": "Придумай 5 слоганов"}]
)

# Кейс 3: Чат-бот (баланс)
response = openai.ChatCompletion.create(
    model="gpt-4",
    temperature=0.7,  # золотая середина
    messages=[...]
)
```

**Другие параметры генерации**

| Параметр | Что делает | Когда использовать
|--|--|--
| top_p (nucleus sampling) | Выбирает токены из топ-p суммарной вероятности | Альтернатива temperature (используют что-то одно)
| frequency_penalty | Штрафует уже использованные токены | Чтобы избежать повторений
| presence_penalty | Штрафует токены из уже упомянутых тем | Для разнообразия тем

```py
# Пример полной конфигурации
response = openai.ChatCompletion.create(
    model="gpt-4",
    temperature=0.8,
    top_p=0.95,
    frequency_penalty=0.5,
    presence_penalty=0.3,
    max_tokens=1000
)
```

### Inference — как модель генерирует ответ

Авторегрессивная генерация
LLM генерирует текст токен за токеном, каждый раз предсказывая следующий токен на основе предыдущих.

```
Шаг 1: "Как" → [предсказание] → "работает"
Шаг 2: "Как работает" → [предсказание] → "градиентный"
Шаг 3: "Как работает градиентный" → [предсказание] → "спуск"
Шаг 4: "Как работает градиентный спуск" → [предсказание] → STOP
```

**Стратегии генерации**

| Стратегия | Описание | Плюсы / Минусы
|--|--|--
| Greedy (temperature=0) | Берём самый вероятный токен | Быстро, детерминированно, но скучно
| Sampling (temperature>0) | Случайно выбираем из распределения | Креативно, но нестабильно
| Beam Search | Держим несколько лучей (кандидатов) | Лучшее качество, но медленно

```py
from transformers import pipeline

generator = pipeline("text-generation", model="meta-llama/Llama-3-8B")

output = generator(
    "Объясни градиентный спуск:",
    max_new_tokens=512,      # сколько новых токенов сгенерировать
    do_sample=True,          # использовать семплирование (иначе greedy)
    temperature=0.7,
    top_p=0.95,
    repetition_penalty=1.1,  # штраф за повторения
    num_beams=1,             # 1 = обычная генерация, >1 = beam search
    early_stopping=True
)
```

**Задержки в продакшене**

| Модель | Размер | Время на токен (GPU A100) | Время на 100 токенов
|--|--|--|--
| Llama 3 8B | 8B | 5-10 мс | 0.5-1 сек
| Llama 3 70B | 70B | 30-50 мс | 3-5 сек
| GPT-4 (API) | ~1.7T | ~50-200 мс | 5-20 сек

**Вывод для инженера**: генерация 1000 токенов может занимать 10-30 секунд. Для реального времени нужно:
* Использовать streaming (отдавать токи по мере генерации)
* Кэшировать KV-cache при повторных запросах
* Использовать speculative decoding (ускорение в 2-3 раза)

### Итог

| Понятие | Простыми словами | Инженерная суть
|--|--|--
| Token | "Кусочек текста" | Единица тарификации и ограничений
| Context window | "Сколько текста модель помнит" | Ограничивает длину промпта + ответа
| Embedding | "Вектор смысла текста" | Используется для поиска и RAG
| Temperature | "Насколько модель фантазирует" | Контролирует случайность
| Inference | "Процесс получения ответа" | Генерируем токен за токеном, медленно


## Prompt Engineering (90% работы инженера)

| Техника | Когда использовать | Пример
|--|--|--
| Zero-shot | Простые задачи, быстрая проверка | `"Классифицируй отзыв: '{text}'"`
| Few-shot | Сложные форматы, редкие задачи | Дать 2-3 примера входа/выхода в промпте
| Chain-of-Thought | Логика, рассуждения, математика | `"Рассуждай по шагам: сначала..., затем..."`
| Role prompting | Стиль, тон, ограничения | `"Ты — эксперт по кредитному скорингу. Отвечай кратко, без воды."`
| Structured output | JSON, XML, строгий формат | `"Ответь ТОЛЬКО в формате JSON: {\"score\": float, \"reason\": str}"`

```py
# Пример: надёжный промпт для классификации
PROMPT = """
Ты — классификатор отзывов на кредитные заявки.

Отзыв: "{review}"

Классифицируй ТОЛЬКО в один из классов:
- "positive" (одобрит)
- "negative" (откажет)  
- "uncertain" (нужен человек)

Ответь СТРОГО в формате JSON:
{{"class": "positive|negative|uncertain", "confidence": 0.0-1.0, "reason": "кратко"}}

Не добавляй ничего кроме JSON.
""".strip()
```

## RAG (Retrieval-Augmented Generation) — главный паттерн

В 90% продакшен-кейсов LLM используется через RAG, а не как "голая" модель.

**Проблема**: LLM не знает ваших документов, данных компании, актуальных цен.

**Решение RAG**:
* **Retrieve** — найти релевантные документы по запросу (через embedding + векторный поиск)
* **Augment** — добавить эти документы в промпт
* **Generate** — LLM отвечает, опираясь на документы

```
Запрос: "Сколько стоит подписка Premium?"
   ↓
Векторный поиск в базе документов
   ↓
Найден документ: "Premium — 999₽/мес"
   ↓
Промпт: "Ответь на вопрос: {запрос}. Используй документ: {документ}"
   ↓
Ответ: "Подписка Premium стоит 999₽ в месяц"
```

Зачем: Дать модели доступ к актуальным/приватным данным без дообучения.

**Когда что использовать: выбор подхода:**
```
┌────────────────────────────────────────────────────┐
│  Задача → Выбор метода                             │
├────────────────────────────────────────────────────┤
│  Простой чат / генерация текста → Zero-shot prompt │
│  Сложная логика / формат → Few-shot + CoT         │
│  Работа с вашими данными → RAG                     │
│  Специфичный стиль / термины → Дообучение (LoRA)   │
│  Жёсткие требования к формату → Функции / Tools    │
│  Критичная точность → Гибрид: RAG + валидация     │
└────────────────────────────────────────────────────┘
```

## Fine-tuning (дообучение) — когда нужно

### RAG vs Fine-tuning:

| Аспект | RAG | Fine-tuning
|--|--|--
| Знания компании | + (через документы) | + (вшивает в веса)
| Актуальность данных | Высокая (документы обновляются) | Низкая (модель frozen)
| Стоимость инференса | Выше (два вызова: поиск + LLM) | Ниже (один вызов)
| Когда выбирать | Часто меняющиеся данные | Стиль ответа, специфичный формат

Типичный fine-tuning: 100-1000 примеров, через API OpenAI или локально (LoRA / QLoRA).


## Оптимизация и деплой

| Метод | Эффект
|--|--
| Кэширование ответов | −30-70% запросов к LLM
| Выбор модели под задачу | GPT-4o-mini для простых, GPT-4o для сложных
| Streaming ответов | Улучшает UX, не снижает cost
| Speculative decoding | Ускорение инференса в 2-4×
| Квантование (INT4/INT8) | −50% памяти, минимальная потеря качества
| Batching запросов | Выше throughput на GPU

**Инструменты для деплоя**

| Задача | Инструменты | 
|--|--
| Локальный инференс | `vLLM`, `TGI` (Text Generation Inference), `llama.cpp`
| Оркестрация | `Kubernetes`, `Ray Serve`, `BentoML`
| API-шлюз | `FastAPI` + middleware для логирования/лимитов
| Мониторинг | `LangSmith`, `PromptLayer`, `Arize`, самописные метрики
| Векторный поиск | `FAISS`, `Qdrant`, `Pinecone`, `Weaviate`


## Метрики для LLM-приложений

| Тип | Метрики | Как собирать
|--|--|--
| Технические | Latency (p50/p99), Error rate, Token usage, Cost per request | Prometheus + логирование
| Качество ответа | Relevance, Faithfulness (не выдумывает), Completeness, Format validity | LLM-as-a-judge, правила, human eval
| Пользовательские | CTR на ответ, "было полезно?", время до следующего запроса | A/B-тесты, фидбэк-кнопки
| Безопасность | Toxicity, PII leakage, Jailbreak attempts | Модерация-модели, правила

**Мониторинг дрейфа и аномалий**

```py
metrics_to_log = {

```

## Инженерные проблемы LLM в продакшене

| Проблема | Решение
|--|--
| Высокая задержка | Streaming ответов, кэширование embedding'ов, более лёгкие модели (Llama 3 8B вместо 70B)
| Галлюцинации | RAG + ограничение токенов + постпроцессинг
| Дорого | Использовать open-source (Llama 3, Mistral, Qwen) на своих GPU
| Дрейф промптов | Версионирование промптов, A/B тестирование
| Безопасность | Валидация входных данных, выходных ответов (фильтры)

## Дообучение — когда промптов и RAG недостаточно

Когда стоит дообучать:
* Нужен специфичный стиль / терминология (медицина, юриспруденция)
* Модель систематически ошибается в вашем домене
* Требуется жёсткий формат вывода, который сложно задать промптом
* Есть 1000+ качественных примеров "вход → желаемый выход"

Когда НЕ стоит:
* "Хочу, чтобы модель лучше понимала мои данные" → сначала RAG
* "Попробуем дообучить, вдруг станет лучше" → сначала few-shot prompting
* Мало данных (< 100 примеров) → overfitting гарантирован

### Методы дообучения (от простого к сложному):

| Метод | Что делает | Когда использовать
|--|--|--
| Prompt tuning / Soft prompts | Обучает "виртуальные токены" в промпте | Мало данных, быстрая адаптация
| LoRA (Low-Rank Adaptation) | Дообучает только маленькие матрицы в слоях | Основной метод для 7-70B моделей
| QLoRA | LoRA + квантование (4-bit) → экономия памяти | Когда мало GPU-памяти
| Full fine-tuning | Обучает все веса | Только если есть ресурсы и ОЧЕНЬ много данных


## Квантование модели

Квантование — это процесс уменьшения точности чисел, которыми модель представляет свои веса и активации. Простыми словами: замена дробных чисел (float32) на более грубые (int8/int4).

### Зачем нужно квантование

| Проблема | Решение через квантование
|--|--
| Модель весит 7 ГБ (Llama 7B в FP16) | → 2-4 ГБ (в Int4)
| Не влезает в память GPU/CPU | → Влезает на дешёвое железо
| Инференс медленный (больше данных → больше операций) | → Ускорение в 2-4 раза
| Дорого (GPU A100 нужна) | → Можно запустить на T4, CPU, даже Raspberry Pi

## Типы квантования (от грубого к точному)

| Тип | Битность | Размер относительно FP32 | Качество | Когда использовать
|--|--|--|--|--
|FP32 (исходная) | 32 бита | 100% | Эталон | Только обучение
|FP16 | 16 бит | 50% | Почти как FP32 | Быстрый инференс на GPU
|BF16 | 16 бит | 50% | Как FP16, но лучше для больших чисел | Альтернатива FP16
|INT8 | 8 бит | 25% | Небольшая потеря (1-2%) | Баланс скорость/качество
|INT4 | 4 бита | 12.5% | Заметная потеря (2-5%) | Максимальная компрессия
|INT2/INT1 | 2-1 бит | 6-3% | Большая потеря | Эксперименты, крайняя экономия

Пример: Llama 7B (FP32) — 28 ГБ → FP16 — 14 ГБ → INT8 — 7 ГБ → INT4 — 3.5 ГБ.

## Как это работает (упрощённо)

```py
# Исходные веса (float32)
weights = [0.123456, -0.789012, 0.345678, ...]

# Квантование в int8: находим min/max и масштабируем
min_val = min(weights)   # -0.9
max_val = max(weights)   # 0.9
scale = (max_val - min_val) / 255  # 0.00706

# Преобразуем в целые числа 0-255
weights_int8 = [int((x - min_val) / scale) for x in weights]
# [100, 12, 145, ...]

# Для инференса: обратное преобразование (дешёвое)
# float_value = min_val + int_val * scale
```

## Основные методы квантования

**Post-Training Quantization (PTQ)** — самый простой. Квантуем модель после обучения, без дообучения.

**Quantization-Aware Training (QAT)** — точнее, но сложнее. Обучаем модель уже с квантованием (симулируем округление во время обучения). Модель учится компенсировать потерю точности.

**GGUF / GGML (для LLM)**. Специализированный формат для LLM (Llama, Mistral, Qwen). Использует квантование + суперблочное сжатие.

**GPTQ / AWQ (для LLM на GPU)**. Более продвинутые методы для GPU, сохраняют лучшее качество, чем GGUF.

## Сравнение методов для LLM

| Метод | Битность | Качество (относительно FP16) | Скорость | Память | Инструменты
|--|--|--|--|--|--
| FP16 | 16 | 100% | 100% | 100% | transformers
| GPTQ | 4 | 97-99% | 110-120% | 25-30% | auto-gptq, exllama
| AWQ | 4 | 97-99% | 110-120% | 25-30% | awq, vLLM
| GGUF (Q4_K_M) | 4 | 96-98% | 80-100% | 25-30% | llama.cpp, Ollama
| INT8 | 8 | 98-99% | 90% | 50% | bitsandbytes, transformers

## Пример: запуск квантованной модели на ноутбуке

```bash
# Установка Ollama (автоматически использует GGUF)
curl -fsSL https://ollama.com/install.sh | sh

# Загрузка квантованной Llama 3 (8B, Q4_K_M)
ollama pull llama3:8b

# Запуск
ollama run llama3:8b
```

Без квантования Llama 3 8B в FP16 требует ~16 ГБ RAM/VRAM. В Q4_K_M — ~5 ГБ. Запускается на любом ноутбуке.

## Трейдоффы, о которых нужно знать

| Аспект | Что теряем | Что выигрываем
|--|--|--
| Точность | 1-5% (для Int4) | 4x уменьшение памяти
| Скорость | На старых GPU — может падать | На новых GPU — растёт за счёт меньших данных
| Сложность | Дополнительный этап конвертации | Возможность запуска на дешёвом железе

*Для LLM: потеря качества при Int4 часто незаметна человеку, а экономия памяти — критична.*

Основные методы:
* PTQ — быстро, но потеря качества
* QAT — точнее, но дорого
* GPTQ/AWQ — лучшее для GPU
* GGUF — лучшее для CPU

**Квантование — это способ запустить большую модель на маленьком железе, пожертвовав небольшим качеством. Для LLM в продакшене это почти всегда необходимо.**
