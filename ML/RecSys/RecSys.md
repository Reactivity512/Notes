# RecSys

## RecSys: алгоритмы, которые нужно знать ML-инженеру

Рекомендательные системы строятся по многоэтапной архитектуре: Retrieval (Recall) → Ranking → Re-ranking. Задача ML-инженера — не просто обучить модель, а сделать так, чтобы она работала за миллисекунды на миллиардах объектов.

```
┌─────────────────────────────────────────────────────────────┐
│              RECSYS ALGORITHMS LANDSCAPE                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🔹 Stage 1: Candidate Generation (Retrieval)              │
│     • Отбор ~100-1000 кандидатов из миллионов               │
│     • Критерий: скорость + recall                           │
│                                                             │
│  🔹 Stage 2: Ranking (Scoring)                             │
│     • Точное ранжирование кандидатов                        │
│     • Критерий: precision + бизнес-метрики                  │
│                                                             │
│  🔹 Stage 3: Re-ranking / Filtering                        │
│     • Бизнес-правила, дедупликация, diversity               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 1.Retrieval (поиск кандидатов)

Здесь нужно за <100 мс из миллионов/миллиардов items отобрать сотню-другую кандидатов. Скорость — критична.

### Коллаборативная фильтрация (Collaborative Filtering (CF))

* **User-based CF**: находит пользователей, похожих на текущего, и рекомендует то, что понравилось им
* **Item-based CF**: находит товары, похожие на те, которые пользователь уже купил или лайкнул
* **Swing** (улучшенный ItemCF): Учитывает не только общие покупки, но и то, насколько уникальны пользователи, купившие оба товара. Эффективен для high-order interactions

**Инженерный нюанс**: Item-based масштабируется лучше — количество товаров обычно стабильнее и меньше, чем количество пользователей. Для вычисления схожести используют cosine similarity или Pearson correlation

### Матричная факторизация (Model-based Collaborative Filtering)

SVD / ALS — классика жанра. Раскладывает огромную матрицу user×item на две матрицы: пользователей и товаров, находя скрытые факторы (например, «любитель ужастиков»)

**Для инженера**: SVD требует периодического переобучения (раз в день/неделю). ALS (Alternating Least Squares) лучше подходит для implicit feedback (клики, просмотры)

### Embedding-based Retrieval (Deep Learning)

**Двухбашенная (Two-Tower / DSSM) модель — индустриальный стандарт на сегодня**

* User Tower: кодирует фичи пользователя и контекста в вектор
* Item Tower: кодирует фичи товара в вектор
* На выходе — косинусная близость или dot product

```
┌─────────────┐     ┌─────────────┐
│ User Tower  │     │ Item Tower  │
│ (features)  │     │ (features)  │
└──────┬──────┘     └──────┬──────┘
       │                   │
       ▼                   ▼
   [embedding]       [embedding]
       │                   │
       └─────► ◄───────────┘
           Dot Product
              │
              ▼
        Similarity Score
```

**Embedding** — это числовое представление объекта (пользователя, товара, слова, видео) в виде плотного вектора фиксированной длины.
Простыми словами: вместо того чтобы описывать объект сотнями разреженных "признаков-флагов" (One‑Hot), мы сжимаем его суть в 50–300 чисел с плавающей точкой.

*Аналогия:*
Представьте координаты на карте: Москва — (55.75, 37.61), *Санкт-Петербург — (59.93, 30.31)*. Это и есть embedding городов. Похожие города окажутся рядом.

**Почему это удобно для инженера**: User и Item векторы можно кэшировать и искать через FAISS (индексация ближайших соседей). Это позволяет делать retrieval из миллионов объектов за миллисекунды

## Ranking (сортировка / точная настройка)

После того как мы отобрали несколько сотен кандидатов, их нужно точно отсортировать. Здесь в дело вступают классические ML-модели: XGBoost, LightGBM, CatBoost. Они отвечают за предсказание CTR (вероятность клика) или конверсии.

Важные модели для признаков (Feature Interaction):
* **FM (Factorization Machines)**: Учитывает взаимодействия признаков (user_id * item_id). Хорошо работает с разреженными данными.
* **DeepFM / DCN**: Комбинируют FM (линейные взаимодействия) и Deep Learning (нелинейности)

**Когда данных очень много - (Neural Ranking Models)**

| Модель | Когда использовать | Сложность
|--|--|--
| Wide & Deep (Google) | Гибридные признаки (sparse + dense) | 3/5
| DeepFM / xDeepFM | Автоматическое обучение feature interactions | 4/5
| DIN / DIEN (Alibaba) | Sequence-aware: учёт истории поведения | 5/5
| Transformer-based (SASRec, BERT4Rec) | Последовательные рекомендации | 5/5

**Важно**: Нейронные ранкеры требуют:
* Огромных данных (10M+ взаимодействий)
* Сложной инфраструктуры (GPU для обучения, оптимизация инференса)
* Тщательного мониторинга (дрейф эмбеддингов)

## Re-ranking / Filtering (Nice to Know)

| Метод | Задача | Пример
|--|--|--
| MMR (Maximal Marginal Relevance) | Баланс relevance + diversity | Не показывать 10 одинаковых чехлов для телефона
| Business Rules | Ограничения бизнеса | Исключить товары без доставки в регион
| Fairness / Debiasing | Устранение позиционного байаса | Не усиливать популярное за счёт нишевого
| Multi-objective Optimization | Баланс метрик | Оптимизировать не только CTR, но и revenue, retention

## Оценка качества: метрики для RecSys

### Retrieval Stage (Candidate Generation)

| Метрика | Что измеряет | Когда использовать
|--|--|--
| Recall@K | Доля релевантных товаров в топ-K | Когда важно не пропустить нужное
| Hit Rate@K | Доля пользователей, у которых есть хотя бы 1 релевантный в топ-K | Бизнес-ориентированная
| MRR (Mean Reciprocal Rank) | Обратный ранг первого релевантного | Когда важен первый результат

### Ranking Stage

| Метрика | Что измеряет | Формула (упрощённо)
|--|--|--
| NDCG@K | Качество ранжирования с учётом позиций | `DCG@K / IDCG@K`, где `DCG = Σ rel_i / log₂(i+1)`
| MAP@K | Средняя точность по всем релевантным | `AP = Σ (Precision@k × rel_k) / total_relevant`
| MRR@K | Средний обратный ранг первого релевантного | `MRR = 1/

```py
# Пример: расчёт NDCG@10
from sklearn.metrics import ndcg_score

# y_true: релевантность товаров (0/1 или градации)
# y_score: предсказанные скоры модели
ndcg_10 = ndcg_score([y_true], [y_score], k=10)
```

### Бизнес-метрики

| Метрика | Как считать | Зачем
|--|--|--
| CTR (Click-Through Rate) | `clicks / impressions` | Базовая метрика вовлечённости
| Conversion Rate | `purchases / clicks` | Качество трафика
| Revenue per Session | `total_revenue / sessions` | Финансовый эффект
| Diversity / Serendipity | Доля уникальных категорий / неожиданных рекомендаций | Долгосрочное удержание

## Как это всё соединяется? (ML Engineering Pipeline)

Рекомендательный сервис — это не одна модель, а конвейер:

1. Retrieval (Выбор кандидатов):
    * Источники: UserCF / ItemCF / Swing / Two-Tower Model.
    * Задача: Достать топ-100 кандидатов из миллиона.
2. Ranking (Точная сортировка):
    * Модель: XGBoost / LightGBM / DeepFM.
    * Задача: Предсказать релевантность каждого кандидата и отсортировать.
3. Re-ranking (Финальная подгонка):
    * Методы: MMR, DPP.
    * Задача: Убрать дубликаты, добавить разнообразия.
