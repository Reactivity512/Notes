# Kafka

Apache Kafka — это распределённая потоковая платформа с открытым исходным кодом, предназначенная для высокопроизводительной обработки данных в реальном времени, обмена сообщениями и организации событийной архитектуры (event-driven). Она обеспечивает горизонтальное масштабирование, отказоустойчивость и хранение потоков сообщений, объединяя компоненты через топики, продюсеры и консьюмеры.

### Зачем нужна Kafka?

**Старый подход (прямые связи)**: Сервис «Клик» напрямую вызывает все остальные сервисы. Если добавится новый сервис (например, «Рекомендации»), придется менять код «Клика». Если «Аналитика» упадет — упадет и «Клик» (или потеряет данные).

**Подход Kafka**: Сервис «Клик» просто кидает событие «Клик по кроссовкам» в Kafka. А все остальные сервисы (Email, Бонусы, Аналитика) сами забирают это событие из Kafka в своем темпе.

**Kafka — это шина данных (Data Bus) или буфер сообщений между генераторами данных (продюсерами) и потребителями данных (консьюмерами).**

## Компоненты Kafka

**Producer** - Отправляет сообщения в Kafka.
**Consumer** - Забирает сообщения из Kafka и обрабатывает их.
**Topic** - Название потока (например, «логи-покупок»).

## Детальная схема работы

Допустим, топик называется `user-clicks`

**1: Producer пишет данные.**
```py
# Псевдокод
producer.send('user-clicks', key='user_123', value='clicked_on_shoes')
```

Сообщение попадает в конец лога (файла на диске) топика.

**2: Как устроен Topic внутри**
Каждый топик делится на Partition (секции) — это как распараллеленные очереди.

Сообщение `user_123` всегда попадает в Partition №0.
Сообщение `user_456` всегда попадает в Partition №1.

Зачем? Чтобы обрабатывать миллион сообщений в секунду — читаем из всех Partition одновременно.

**3: Consumer читает данные**
Consumer знает, откуда читать, благодаря Offset (смещение) — это порядковый номер сообщения в Partition (0, 1, 2, 3...).

Consumer говорит: *«Дай мне сообщение с offset=5 из Partition 0»*.

**4: Группа потребителей (Consumer Group)**
Если у вас 5 сервисов «Аналитики», они объединяются в группу `analytics-group`. Kafka автоматически распределит Partition между ними: один читает Partition 0, другой — Partition 1 и т.д.

## Главные "фишки" Kafka

* Хранение на диске (не в RAM). Kafka пишет прямо в файлы на жестком диске. Благодаря последовательной записи и кэшу ОС, это работает так же быстро, как память, но дешевле и надежнее.

* Отказоустойчивость (Replication). Каждый Partition имеет копии (лидер и последователи). Если сервер сломался — лидером становится другой сервер за секунду.

* Вечность данных. Вы не теряете данные, если Consumer отключился. Когда он вернется, он прочитает всё, что накопилось с момента его отключения.

* Воспроизведение (Replay). Можно сбросить offset на 0 и обработать все сообщения за последний месяц заново (например, после исправления бага в алгоритме).

## Когда используют Kafka?

| Сценарий | Почему именно Kafka |
|--|--|
| Логи от 1000 серверов | Огромная пропускная способность (миллион сообщений/сек) |
| Банковские транзакции | Сообщения не теряются (гарантия durability) |
| Отслеживание заказа курьера | Нужно хранить историю перемещений (не просто последнюю точку) |
| CDC (Change Data Capture) | Перехват изменений в базе данных PostgreSQL → отправка в ElasticSearch |
| Микросервисная архитектура | Нужно связать 30 микросервисов без жесткой связанности |


## Полная схема: от отправки до чтения

```
[Producer]                [Kafka Broker]                    [Consumer]
    |                           |                                 |
    |-- 1. Запрос метаданных --> |                                 |
    |<-- 2. Лидер партиции ----  |                                 |
    |                           |                                 |
    |-- 3. Отправка данных ----> |                                 |
    |                           | (4. Запись на диск)             |
    |                           | (5. Replication)                |
    |<-- 6. Подтверждение (ack)- |                                 |
    |                           |                                 |
    |                           |<-- 7. Poll (запрос данных) ----|
    |                           |                                 |
    |                           |-- 8. Данные + offset --------->|
    |                           |                                 |
    |                           |<-- 9. Commit offset -----------|
```

### 1. Что происходит ДО отправки (инициализация)
Прежде чем отправить первое сообщение, Producer должен узнать, куда именно писать.

```py
producer = KafkaProducer(bootstrap_servers=['broker1:9092', 'broker2:9092'])

# Что происходит внутри:
# 1. Producer подключается к любому брокеру
# 2. Спрашивает: "Где находится лидер партиции для топика 'orders'?"
# 3. Брокер отвечает: "Топик 'orders' имеет 3 партиции:
#    - Partition 0: лидер на broker2
#    - Partition 1: лидер на broker1
#    - Partition 2: лидер на broker3"
```

Зачем это нужно? Kafka не хранит весь топик на одном сервере. Данные распределены. Producer должен писать напрямую в лидер-брокера нужной партиции.

### 2. Выбор партиции
Если вы не указали ключ сообщения, Kafka сама решает, в какую партицию писать (round-robin или sticky partitioner).

Round-Robin Partitioner
Распределяет сообщения по кругу между всеми партициями. Равномерная нагрузка, порядок не важен.
```
Сообщение 1 → Partition 0
Сообщение 2 → Partition 1
Сообщение 3 → Partition 2
Сообщение 4 → Partition 0 (снова)
```

Sticky Partitioner
Привязывает сообщения к одной партиции для batch-отправки. Меньше запросов, выше throughput, меньше latency.
```
Сообщения 1-100 → Partition 0 (batch)
Потом переключается → Partition 1 на следующий batch
```

Пример:
```py
# Сообщения с одним user_id попадают в одну партицию
producer.send('orders', key='user_123', value='order1')  # -> Partition 0
producer.send('orders', key='user_123', value='order2')  # -> Partition 0 (тот же!)
producer.send('orders', key='user_456', value='order1')  # -> Partition 1
```

### 3. Producer отправляет данные

1.1 Producer batch (аккумуляция)
Producer не отправляет каждое сообщение отдельно — это слишком дорого (сетевые вызовы). Он накапливает их в пачки (batches)

```py
# Настройки Producer
batch_size = 16384  # 16KB - размер пачки
linger_ms = 5       # ждать до 5 мс, даже если пачка не полная

# Внутренний буфер Producer:
# [msg1, msg2, msg3, msg4] -> упаковал в batch -> отправил
```

Таймлайн:
```
t=0ms:  Пришло msg1 -> кладем в буфер для Partition 0
t=2ms:  Пришло msg2 -> кладем туда же
t=5ms:  Таймер сработал (linger_ms) -> отправляем batch из 2 сообщений
```

1.2 Сжатие (compression)
Перед отправкой Producer может сжать batch:
```py
compression_type = 'snappy'  # или 'gzip', 'lz4', 'zstd'

# Без сжатия: 100 сообщений по 1KB = 100KB по сети
# Со snappy: ~30-40KB (экономия 60-70%)
```

1.3 Отправка и подтверждения (acks)
Producer отправляет batch лидеру партиции и ждет подтверждения. Уровень надежности задается параметром `acks`:

| acks | Что значит | Риск | Скорость |
|--|--|--|--|
|0 | "Отправил и забыл" | Потеря данных (высокий) | Максимальная |
|1 | Лидер записал на диск → ОК | Потеря при падении лидера до репликации | Высокая |
|all (-1) | Все реплики (ISR) записали → ОК | Почти нулевой | Минимальная |

```py
# Пример с acks=all
producer.send('orders', value=b'order_data', acks='all')
# Producer ждет, пока ВСЕ реплики (минимум 2 из 3) подтвердят запись
```

### 4. Брокер записывает на диск

Это самое важное и неочевидное. Kafka не пишет случайно, она пишет последовательно.

4.1 Структура на диске
```bash
/var/kafka/data/
├── orders-0/          # Partition 0 топика orders
│   ├── 00000000000000000000.log    # файл с данными (сегмент)
│   ├── 00000000000000000000.index  # индекс (смещения -> позиции)
│   ├── 00000000000000000000.timeindex
│   ├── 00000000000000100000.log    # следующий сегмент (100000 записей)
│   └── 00000000000000100000.index
├── orders-1/          # Partition 1
└── orders-2/          # Partition 2
```

4.2 Запись в файл (append-only)
Когда приходит batch:
```c
// Псевдокод брокера
FileSegment current_segment = open("/data/orders-0/00000000000000000000.log");

// Просто дописываем в КОНЕЦ файла (append)
current_segment.append(record_batch);  // операция O(1)

// Обновляем индекс: offset 12345 -> позиция в файле 876543
index.write(offset, position);
```

Почему это быстро? Потому что жесткие диски любят последовательное чтение/запись, но ненавидят случайное. Kafka использует это свойство.

4.3 Страничный кэш ОС (Page Cache)
Kafka не пишет напрямую на диск через `fsync`. Она пишет в кэш страниц операционной системы:
```
Producer -> [Kafka Broker] -> [Page Cache (RAM)] -> [Disk (фоном)]
                                    |
                                    +--> [Consumer] (читает из кэша, если данные там)
```

ОС сама решит, когда сбросить на диск (обычно через несколько секунд). Это дает:
* Огромную скорость (работа с RAM, а не с диском)
* Бесплатное кэширование (не нужно отдельный кэш)

Но есть риск: при внезапном отключении электричества последние секунды данных могут потеряться (если acks=1).

### 5-6. Репликация, Подтверждение

После записи на лидере, репликация происходит асинхронно, но с гарантиями.

5.1 ISR (In-Sync Replicas)
Брокеры следят за отставанием реплик:
```yaml
replica.lag.time.max.ms=10000  # если реплика не ответила 10 секунд - исключить из ISR
min.insync.replicas=2          # минимум 2 реплики в ISR для acks=all
```

Пример с 3 брокерами (replication.factor=3):
```
Partition 0: Лидер = Broker1
             ISR = [Broker1, Broker2, Broker3]  # все три в синхронизации

Broker2 (follower) постоянно тянет данные от Broker1:
    follower.fetch.thread -> "Дай мне все новые сообщения"
```

Если Broker2 отстал >10 секунд:
```
ISR = [Broker1, Broker3]  # Broker2 вылетел из ISR
```

5.2 Синхронизация шаг за шагом
```py
# Шаг 1: Producer пишет на лидер (Broker1) с acks=all
broker1.append(record)  # запись в page cache

# Шаг 2: Лидер ждет подтверждения от ISR
wait_for(min_insync_replicas - 1)  # 2 реплики должны подтвердить

# Шаг 3: Реплики (Broker2, Broker3) подтверждают
broker2.ack()  # "Я скопировал"
broker3.ack()  # "Я скопировал"

# Шаг 4: Лидер отвечает Producer
producer.ack()  # "Успех, данные сохранены"
```

### 6-7. Consumer читает данные (poll loop)

Consumer работает по принципу pull (вытягивает), а не push (ему толкают).

Consumer запрашивает данные
```py
consumer = KafkaConsumer('orders', group_id='order-processor')
records = consumer.poll(timeout_ms=1000)  # забрать все доступные сообщения
```

### 8. Данные + offset,
Внутри consumer.poll()

```
Consumer -> Брокер (лидер партиции): "Дай мне данные из Partition 0, начиная с offset 500"

Брокер:
1. Ищет файл, где лежит offset 500 (через индекс)
2. Читает из файла, начиная с нужной позиции
3. Отправляет batch сообщений (до max.poll.records)

Consumer получает данные (НЕ удаляет их из Kafka)
```

### 9. Commit offset

Consumer сам решает, когда "сохранить прогресс":
```py
# Автокоммит (опасно, может потерять данные)
enable_auto_commit=True
auto_commit_interval_ms=5000  # каждые 5 секунд сохранять offset

# Ручной коммит (надежно)
records = consumer.poll()
for record in records:
    process(record)
consumer.commit()  # теперь offset сохранен
```

Если Consumer не скоммитил offset и упал:
* При перезапуске он прочитает последний скоммиченный offset
* Данные между последним коммитом и падением будут обработаны повторно

Если Consumer скоммитил offset, но не обработал данные (**баг**):
* Данные потеряны (их уже не прочитать, если retention истек)

## Полный пример с цифрами

Допустим, топик orders с 3 партициями, retention = 7 дней.

```py
# Настройки
batch_size = 16384 (16KB)
linger_ms = 5
compression = snappy
acks = all

# Отправляем 1000 сообщений по 100 байт
for i in range(1000):
    producer.send('orders', key=f'user_{i%10}', value=f'order_{i}')
```

Что происходит:
* Сообщения группируются по партициям (по хэшу ключа)
* Накапливаются в batch по 16KB (~160 сообщений)
* Каждые 5 мс или при заполнении batch - отправка
* Брокер получает ~6-7 batch'ей (1000/160)
* Каждый batch пишется в конец файла партиции
* Реплики копируют данные
* Producer получает подтверждение

Consumer читает:
```py
consumer = KafkaConsumer('orders', group_id='processor', 
                        max_poll_records=500,
                        enable_auto_commit=False)

while True:
    records = consumer.poll(timeout_ms=1000)
    # records = {0: [msg1, msg2, ...], 1: [msg100, ...]}
    
    for partition, messages in records.items():
        for msg in messages:
            save_to_database(msg)  # бизнес-логика
        # После успешного сохранения - коммитим
        consumer.commit({partition: messages[-1].offset})
```

Что происходит:
* Consumer запрашивает до 500 записей за раз
* Брокер читает из page cache (очень быстро)
* Consumer обрабатывает и коммитит offset
* Если Consumer упал после коммита, данные не потеряны
* Если упал до коммита - данные будут прочитаны повторно


## Что происходит при сбоях?

### Сценарий 1: Лидер партиции упал

```
Было: Partition 0: Leader=Broker1, ISR=[Broker1, Broker2, Broker3]
Broker1 упал (выключили сервер)

Что делает Kafka:
1. Контроллер кластера (один из брокеров) замечает потерю heartbeat
2. Выбирает нового лидера из ISR (например, Broker2)
3. Обновляет метаданные (Producer'ы и Consumer'ы узнают об этом при следующем запросе)
4. Broker2 становится лидером

Время: ~1-5 секунд (зависит от настроек)
```

### Сценарий 2: Consumer упал

```
# Consumer с group_id='processor' упал на полпути
# (обработал 300 из 500 сообщений, коммит не сделал)

# Через 10 секунд (session.timeout.ms) consumer перестал считаться живым

# Kafka делает перебалансировку (rebalance):
# Partition 0 назначается другому consumer'у в той же группе

# Новый consumer начинает читать с ПОСЛЕДНЕГО скоммиченного offset
# (например, offset 200, а было обработано 300)

# Результат: сообщения 201-300 будут обработаны ПОВТОРНО
```

## Вывод:
* Producer пишет в конец файла (append-only) → поэтому быстро
* Consumer читает из любого места (по offset) → поэтому можно перечитывать
* Данные не удаляются после прочтения → живут по retention
* Offset управляет consumer → он решает, что прочитать
* Репликация + acks=all → надежность, но медленнее
* Page cache → скорость RAM, персистентность диска
