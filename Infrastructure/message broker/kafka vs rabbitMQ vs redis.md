# Kafka vs RabbitMQ vs Redis


| Характеристика | Kafka | RabbitMQ | Redis|
|--|--|--|--|
|Модель| Distributed log (pub/sub с persistence) | Message queue + routing | In-memory stream + pub/sub|
| Основная задача | Потоковая обработка данных, лог событий | Классическая очередь задач | Кэш, база данных в памяти |
| Удаление сообщений | Через время (дни/часы) или по размеру | Сразу после обработки | По умолчанию сразу, можно настроить|
| Макс. размер сообщения | 1 МБ (можно увеличить до 100 МБ) | ~512 МБ | ~512 МБ |
| Задержка | Миллисекунды (2-20 мс) | Микросекунды (до 2 мс) | Наносекунды (0.2 мс) |
| Пропускная способность | Огромная (1 млн+ сообщений/сек) | Средняя (50-100 тыс/сек) | Огромная (1 млн+)|
| Хранение данных | На диске (долговременно) | В RAM + диск (выгружается) | В RAM (основное) + диск (опционально)|
| Перечитывание старых данных | Да (основная фича) | Нет (удалилось) | Нет (удалилось)|
| Масштабирование | Горизонтальное (partitions) | Вертикальное + clustering | Вертикальное + Redis Cluster|


## Kafka vs RabbitMQ

**Аналогия:**

*RabbitMQ* = Почтальон: Забрал письмо, доставил. Если вы не дома — письмо вернется на почту (очередь), но не навсегда.

*Kafka* = Архивная служба: Каждое письмо подшивается в папку (лог). Вы можете прийти через месяц и перечитать его снова. И другие сотрудники тоже могут читать ту же папку параллельно.

| Аспект | RabbitMQ | Kafka |
|--|--|--|
| Модель удаления | Подтверждение (ACK) → удаление | Никто не удаляет, истекает TTL |
| Маршрутизация | Гибкая (exchanges, bindings, routing keys) | Только топики (жесткая структура) |
| Приоритеты задач | Да (очереди с приоритетом) | Нет |
| Отложенные сообщения | Да (TTL + DLX) | Нет (нужен отдельный сервис) |
| Consumer группы | Одна очередь — один потребитель (или конкурентная борьба) | Много групп читают тот же топик независимо |
| Когда выбирать | Веб-приложения, фоновые задачи, RPC, где важна маршрутизация | Аналитика, логи, CDC, когда данные нужно хранить и перечитывать |

Пример выбора:
* *RabbitMQ* → отправка email после регистрации, обработка загруженного фото, распределение заданий между воркерами.
* *Kafka* → сбор кликов всех пользователей за день, передача изменений из БД в поиск, потоковая аналитика через Spark/Flink.

## Kafka vs Redis

**Аналогия:**

*Redis* = Записная книжка в кармане: Супер-быстрая, но маленькая (вся в оперативной памяти). Если выключить свет — всё пропало (если не настроен диск).

*Kafka* = Библиотечный архив: Медленнее, но огромный (на жестких дисках). Данные переживут перезагрузку.

| Аспект | Redis | Kafka|
|--|--|--|
| Хранение | RAM (быстро, мало) | Диск (медленнее, много) |
| Структуры данных | Строки, списки, хэши, битмапы, HyperLogLog | Только последовательности байтов (массив) |
| Гарантии доставки | At-most-once (сообщение доставлено 0 или 1 раз. Потеря возможна.) | At-least-once (сообщение гарантированно доставлено, но возможны дубликаты) или exactly-once (сообщение доставлено ровно один раз, без дубликатов) |
| Pub/Sub | Есть, но сообщения не хранятся (подписчик отключился → потерял) | Хранятся всегда (подписчик получит при переподключении) |
| Персистентность | Опционально (RDB/AOF) | Всегда (по умолчанию) |
| Типичная задача | Кэш, сессии, лидерборды, счетчики | Потоки событий, ETL, интеграция систем |

```py
# Redis: Если подписчик отвалился на 1 секунду - сообщение потеряно навсегда
redis.publish('chat', 'hello')  # никто не слушает -> пропало

# Kafka: Если консьюмер отвалился, он прочитает при возвращении
producer.send('chat', 'hello')  # сообщение сохранится на диске
```

Когда Redis подходит как очередь:
* Вам нужна микросекундная задержка (игровой сервер, чат)
* Данные не критичны (можно потерять пару сообщений)
* Очередь небольшая (все влезает в RAM)

Когда Redis НЕ подходит:
* Нужно надежно хранить миллионы сообщений (убивает RAM)
* Консьюмеры могут отваливаться (потеряете данные)
* Нужно перечитать сообщение заново

## Критические различия в гарантиях

| Гарантия | Kafka | RabbitMQ | Redis |
|--|--|--|--|
| Сохранность при падении продюсера | Да (ack от брокера) | Да (confirm mode) | Нет (lost on crash) |
| Сохранность при падении консьюмера | Да (offset не коммитится) | Да (no ack) | Нет (данные в памяти) |
| Порядок сообщений | В пределах партиции | Гарантирован для очереди | Нет (список — да, pub/sub — нет) |
| Deduplication | Idempotent producer (это режим работы отправителя сообщений, гарантирующий доставку данных ровно один раз («exactly-once») в рамках одного раздела (partition), предотвращая дубликаты при сбоях сети или брокера. Он автоматически присваивает уникальный ID производителю и номер каждому сообщению) | Нет | Нет |

Дедупликация данных (Data Deduplication) — это технология оптимизации хранения, которая устраняет избыточные копии информации, сохраняя только один уникальный экземпляр данных.
